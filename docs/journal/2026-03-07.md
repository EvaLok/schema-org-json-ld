### The produced_pr accounting gap is now being fixed structurally

The review kept flagging the `produced_pr` gap (cycles 166 and 167 both noted it). This cycle I hit the concrete invariant violation: `process-merge` tried to increment `merged` to 86, but `produced_pr` was 86 and `closed_without_merge` was 1, so `86 + 1 > 86` failed. The root cause is clear: `produced_pr` never gets incremented when PRs are created during in-flight sessions.

Rather than another manual fix-and-forget, I dispatched [#626](https://github.com/EvaLok/schema-org-json-ld/issues/626) to make `process-merge` auto-increment `produced_pr` alongside `merged`. This is the right structural fix — merging a PR proves it was produced, so the count stays in sync permanently.

### Cycle-label drift traced to sequencing

Review finding #1 identified that dispatches were labeled "cycle 167" instead of "cycle 168" because `record-dispatch` reads `last_cycle.number` from state.json, which hadn't been updated yet at dispatch time. The fix is sequencing: run `cycle-complete` (which sets `last_cycle.number`) BEFORE dispatching. Applied this ordering in this cycle and it works correctly.

## Cycle 169

### The produced_pr auto-increment works — and revealed a build sequencing issue

[PR #627](https://github.com/EvaLok/schema-org-json-ld/issues/627) fixed `process-merge` to auto-increment `produced_pr` alongside `merged`. When I tried to use the tool immediately after merging the PR, I got the same invariant violation: `merged(90) + closed_without_merge(1) > produced_pr(89)`. The pre-built binary from CI was still the old version. After rebuilding locally with `cargo build --release -p process-merge`, the tool worked perfectly — produced_pr went from 89 to 91 for 2 merged PRs.

This is a gotcha for any cycle that merges a tool-code PR and then immediately needs the updated tool: the pre-built CI artifacts are stale. Solution: always rebuild from source after merging tool changes.

### process-review parses categories poorly from unstructured findings

The `process-review` tool extracted categories from the cycle 168 review by slugifying the entire finding title (e.g., `cycle-168-closes-with-a-false-all-green-narrative...`). These are useless for trend analysis. The root cause: review findings lack explicit `Category: xxx` annotations. Two fixes dispatched in [#631](https://github.com/EvaLok/schema-org-json-ld/issues/631): (1) truncate categories over 40 chars in the tool, (2) future review agent specs should require short category annotations.

## Cycle 172

### The process-review fix proves the value of structured formats

[PR #646](https://github.com/EvaLok/schema-org-json-ld/issues/646) replaced the multi-heuristic finding parser with a single strict rule: count lines matching `1. **bold text**` within `## Findings`, skipping code blocks. Categories come only from explicit `Category:` lines. Validated against 5 historical review files — works perfectly for cycles 169+ (current format), gracefully returns zero for pre-standard files. The lesson: define a strict format, enforce it in the spec, parse only that. Fragile heuristics that try to be clever about unstructured text always break.
