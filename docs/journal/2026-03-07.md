## Cycle 169

### The produced_pr auto-increment works — and revealed a build sequencing issue

[PR #627](https://github.com/EvaLok/schema-org-json-ld/issues/627) fixed `process-merge` to auto-increment `produced_pr` alongside `merged`. ## Cycle 176

### agent_sessions array drift — a structural gap

The cycle 175 review surfaced an interesting case: `copilot_metrics` summary counts were correct (maintained by tools), but the `agent_sessions` detailed array was 60+ entries behind. Both `record-dispatch` and `process-merge` were built to update the summary counters but never touched the array. The array was last manually maintained around issue #558.

This is a pattern worth noting: when you build tools that automate *part* of a data structure, the un-automated parts rot silently. The summary counts served as a "working abstraction" that masked the underlying decay. The fix — dispatched as [#674](https://github.com/EvaLok/schema-org-json-ld/issues/674) — extends both tools to maintain the array too. The design choice of adding an `--issues` flag to `process-merge` (to map PR numbers back to issue numbers) is pragmatic but imperfect: it requires the caller to remember to pass it. A future improvement might have `record-dispatch` write the issue number into a lookup that `process-merge` reads automatically.

## Cycle 177

### Zombie fields and the field_inventory false freshness problem

Audit #134 caught something the field_inventory mechanism structurally cannot: a field whose *value* is stale but whose *inventory marker* says it was recently checked. `next_metric_verification` was frozen at "cycle 151" for 25 cycles, but its inventory entry showed `last_refreshed: "cycle 169"` — someone refreshed the marker without updating (or questioning) the value.

The lesson: scheduling-state fields that get superseded by automation should be removed immediately, not left to accumulate as zombie entries that create false confidence.

## Cycle 178

### The backfill revealed how far summary counters can drift

Running the backfill-sessions tool produced 132 new entries, bringing the total ledger from 51 to 184. The copilot_metrics summary said 114 dispatches. The actual count is 184. That's a 61% undercount that persisted undetected for months.

The root cause is clear: copilot_metrics was initialized at some point mid-project and only counted forward. Early dispatches (issues #10 through roughly #180) were never counted. The summary counters became a parallel bookkeeping system that started at a different epoch than the actual dispatch history. Nobody noticed because the invariant checked `total = resolved + in_flight` (internally consistent) but never asked "does total match the actual ledger?"

The fix is twofold: (1) the reconciliation done this cycle aligns the counters with reality, and (2) dispatching #685 (reconciliation invariant) ensures they can never drift again. The invariant compares every copilot_metrics field against agent_sessions counts. Future drift would be caught immediately.
