# Journal — 2026-03-03

## Cycle 104: When a threshold becomes a uniform rule

**Eva's [#340](https://github.com/EvaLok/schema-org-json-ld/issues/340) resolves a design tension the audit identified but didn't act on.** The audit journal noted constructor mismatches during QC parity testing — ItemList and ContactPoint using positional params that confused the QC's test authors. The audit correctly classified these as "test construction issues, not library bugs." True, but incomplete: the underlying problem is API ergonomics, not test code quality.

The original convention (audit #20, accepted cycle 72) was: options-object for types with >5 optional properties, positional for simpler types. This created a dual pattern — consumers had to know which pattern each class used. PHP doesn't have this problem because named parameters make all constructors position-independent. TypeScript's lack of named parameters means the only way to match PHP's ergonomics is to use options objects everywhere.

**Eva saw what the orchestrator and audit didn't.** The threshold rule optimised for code size (positional params are more concise). Eva optimised for consumer experience. `new Brand({ name: 'ACME' })` is two characters longer than `new Brand('ACME')` but eliminates a category of error: passing arguments in the wrong order. When you're writing `new ContactPoint('+1-800-123-4567', null, null, null, 'en')` to set only telephone and language, the options pattern isn't a style preference — it's a usability fix.

**This is a breaking change and that's fine.** The package isn't published yet. The QC constructor mismatches that triggered this observation exist only in QC test code, not in external consumers. This is exactly the right time to standardise: after the port is complete but before the first release. Once published, changing constructor signatures requires a major version bump.

**Decomposition strategy:** 3 sequential batches (A-D, E-M, O-V). Sequential is mandatory per audit #29 (barrel file conflicts). The refactoring is mechanical — extract positional params to options interface, update constructor body, update tests. The agent has handled similar batch sizes (9-12 types) during the TS port. 16 classes per batch should be comfortable.

**Note on audit #56:** QC-targeted, about the QC's Copilot dispatch pipeline. No action needed from main orchestrator. The audit is now filing recommendations about the QC's agent workflow — a sign that the QC is evolving from a pure validation tool to a development participant.

**Update (01:27Z):** All 3 batches completed and merged within a single cycle (~40 minutes wall clock). 53 classes converted, zero revision rounds needed from any of the 3 Copilot sessions. The mechanical nature of the refactoring made it ideal for agent delegation — the pattern is regular, the rules are clear, and the test suite catches any mistakes. This is the largest single-cycle code change in the project's history: ~160 files across 3 PRs. Eva directive #340 closed.

## Cycle 106: The parity denominator gap returns

This is a variant of the denominator problem the audit caught in [#49](https://github.com/EvaLok/schema-org-json-ld-audit/issues/49). The mechanism is different but the risk is the same: the QC could reach 86/86 E2E while parity stays at 39/86, creating a misleading completion signal.

The root cause is that the QC's Copilot dispatch specs include E2E generate scripts and unit tests, but not parity tool updates. Two process fixes are possible: (1) add a post-merge step to the QC's PR review pipeline, or (2) include parity test creation in the Copilot dispatch spec. Either works; the key requirement is that parity advances in lockstep with E2E.

## Cycle 107: Systematic property gap auditing as a maintenance practice

**The property gap audit is now systematic.** Cycle 106 scanned 5 types and found 2 actionable gaps. This cycle scanned 10 more types in two parallel batches and found the coverage picture is remarkably good across most types, with a few targeted gaps worth fixing.

**Coverage scorecard (15 types audited total):**
- **100% covered (10 types):** Article, Recipe, Event, Product, FAQ, Organization, LocalBusiness, SoftwareApplication, Movie, ProfilePage (class itself)
- **Near-complete with targeted gaps (5 types):** VideoObject (missing SeekToAction), Course (deprecated Google docs), Dataset (missing hasPart/isPartOf), JobPosting (QuantitativeValue.unitText), DiscussionForum (type mismatches)

**The SeekToAction gap is interesting.** It's a recommended property for Key Moments seek markup — the feature that lets Google deep-link into specific video timestamps. The codebase already has the exact pattern needed (MathSolver + SolveMathAction with PROPERTY_MAP for hyphenated JSON-LD keys). This is a great example of how incremental auditing reveals gaps that aren't obvious from the schema.org spec alone — you need the Google docs to know that `potentialAction` with `SeekToAction` is the recommended way to implement seek markup.

**QuantitativeValue.unitText vs unitCode.** A subtle distinction: `unitCode` uses UN/CEFACT machine codes (KGM, LBR), while `unitText` is the human-readable equivalent (HOUR, YEAR). Google's JobPosting docs specifically use `unitText` for salary periods. We have the former but not the latter. Non-breaking addition, but important for correctness.

**Course's 12 missing properties are a false alarm.** The `/course-info` Google docs page was deprecated in September 2025. The active `/course` page only formally requires `name` and `description` with `provider` recommended. The missing properties were from the deprecated page. This is a good reminder to verify the current state of Google's documentation, not just the cached knowledge.

**Audit #60 (QC dispatch metrics) is primarily QC-targeted.** The main orchestrator has 21/21 dispatch-to-PR conversion — no failures. But the audit's recommendation to track `dispatch_to_pr_rate` separately from `pr_merge_rate` is sound defensive practice. Added both metrics to state.json and stale dispatch detection to the startup checklist.

**Observation on the QC pipeline.** QC parity is at 49/86 (57%), up from 39/86 at cycle 106. The expansion rate is ~10 types per QC cycle. At this rate, reaching 86/86 will take ~4 more QC cycles. Meanwhile, each main orchestrator cycle is now primarily a maintenance cycle — property gap auditing, infrastructure refinement, and audit processing. This is fine; the rate-limiting factor is the QC, and rushing won't help.

## Cycle 108: The mixed-array bug and type classification

**First revision round in 23 dispatches.** PR [#364](https://github.com/EvaLok/schema-org-json-ld/issues/364) exposed a pre-existing bug in `JsonLdGenerator::AddPropertiesToObject()`: it checked only the first array element's type to decide serialization strategy for the entire array. When `Dataset.hasPart` contained `[Dataset object, "https://url"]`, the first element was TypedSchema so all elements got passed to `SchemaToObject()`, crashing on the string.

The bug has been latent since the JsonLdGenerator was written — no existing property used mixed arrays before `Dataset.hasPart`. Interestingly, the TypeScript serializer already handled this correctly by dispatching per-element. The PHP code was checking `$v[0] instanceof TypedSchema` and treating the result as gospel for all elements. Copilot fixed it in 4 minutes by flattening the branching into a single loop with per-element type checking.

**Lesson:** The PHP serializer's assumption of array homogeneity was never tested because all existing array properties were homogeneous. The test-driven approach (write the expected behavior, then make it work) is what caught this — without the mixed-array test case, the bug would have shipped. This validates the importance of expanding property types beyond what's been tested historically.

**Type classification was high-value.** Audit [#62](https://github.com/EvaLok/schema-org-json-ld-audit/issues/62) correctly identified that the QC orchestrator's coverage expansion pipeline was dispatching building-block types for standalone E2E testing. These types are designed to be nested — standalone validation will always fail.

The classification I posted on QC-REQUEST [#331](https://github.com/EvaLok/schema-org-json-ld/issues/331) splits the 88 schema types into ~28 standalone-testable, ~58 building-block, and 12 enums. This reframes the QC's publish gate: E2E testing should target the 28 standalone types, not all 88. Parity testing should still cover all 88. The QC orchestrator can use this to skip building-block types in its expansion pipeline, avoiding wasted Copilot sessions.

**Property gap audit is now complete.** All 23 standalone-testable types have been audited against Google's current documentation. 16/23 at 100% coverage. The remaining gaps are low-priority: QAPage's `Question.image`/`video` and `Answer.image`/`video` (recommended, not required), and VacationRental's `Review.contentReferenceTime` (French listings only). Every actionable gap found in cycles 106-108 has been either fixed (PRs #355, #357, #362, #364) or dispatched (#370).

**AggregateRating.itemReviewed is a legitimate library enhancement.** The QC report [#160](https://github.com/EvaLok/schema-org-json-ld-qc/issues/160) wasn't a bug in the traditional sense — AggregateRating works correctly when nested. But Google requires `itemReviewed` for standalone validation. Adding it as an optional property (defaulting to null) maintains backward compatibility while enabling standalone use. The pattern follows Review's existing `itemReviewed: TypedSchema|null`.
