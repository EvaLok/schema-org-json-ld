The [PR #2706](https://github.com/EvaLok/schema-org-json-ld/issues/2706) semantic bug should have been caught at dispatch time. [Audit #439](https://github.com/EvaLok/schema-org-json-ld-audit/issues/439) direction 1's spec assumed login-based detection would work, but did not enumerate the orchestrator's actual GitHub identity in this repo. The agent followed the spec faithfully and unit-tested with mocked logins. Live invocation immediately surfaced the gap. This is exactly the failure mode that directive [#809](https://github.com/EvaLok/schema-org-json-ld/issues/809) (do not merge with known issues) catches — the iterate-until-ready discipline held, but the upstream spec gap that produced the bug was avoidable. Lesson: tools that compare 'who posted X' must use signature-based detection in this repo, since EvaLok is the shared GitHub identity for both human-Eva and orchestrator-PAT. AGENTS.md addition would crystallize this for future dispatches.

The cycle 538 review F3 (standing Eva blockers list reintroducing closed/responded issues) was deferred for capacity reasons but is also distinct from any in-flight fix. It needs its own dispatch in a future cycle: the journal-rendering logic should derive the blockers list from the open question-for-eva set + last-Eva-response status (now that [#2706](https://github.com/EvaLok/schema-org-json-ld/issues/2706) will provide the latter once revised). Currently the blockers list is hand-maintained in each journal entry, and the recurring 'list closed issues as standing blockers' pattern is exactly what cycle 538 review F3 caught.
