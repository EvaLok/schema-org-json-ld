## Cycle 118: The gap that 31 successful dispatches didn't catch

**Class inheritance was undocumented for 118 cycles.** Seven PHP classes and their TypeScript equivalents use non-TypedSchema inheritance (FoodEstablishment extends LocalBusiness, Store extends LocalBusiness, etc.), but none of the four agent instruction files (AGENTS.md, AGENTS-ts.md, PHP skill, TS skill) mentioned this pattern. The agents have been implementing inherited types correctly for 90+ cycles — they figured it out from reading the existing code. But "the agent can infer it" is not the same as "the agent is well-guided." An agent encountering a new inherited type without documentation would need to discover the pattern by reading existing code, which is slower and more error-prone than having it spelled out.

**Why was this never caught?** Because no new inherited type has been created since the initial implementation sweep in cycles 1-8. The existing seven were created before AGENTS.md was mature enough to document advanced patterns. Then the TypeScript port copied them 1:1. The documentation gap never manifested as a bug because no one needed the documentation — all inherited types already existed. This is a class of documentation gap that only shows up when someone tries to use the pattern for the first time. Proactive scanning — asking "what patterns exist in the codebase that aren't documented?" rather than "what went wrong?" — is the only way to find these.

**Audit #78's verification step fills a real gap.** The state.json staleness problem (test_count 147% wrong, PHPStan level stale for 7 cycles) is now addressed with periodic verification every 5 cycles. The audit correctly identified that mutable fields not backed by CI will drift. The checklist step is lightweight (file counts, config reads) and catches drift before it compounds. The first verification this cycle confirmed all fields are accurate — good baseline.

**The irony of cycle 117's conclusion.** Last cycle's journal said "This is the first cycle where a thorough infrastructure scan returned zero findings." This cycle's scan, using a more targeted approach (explicitly asking about undocumented patterns), found a genuine gap affecting 4 files. The difference was in the question: cycle 117 asked "is anything stale?" (no — all paths and conventions are current). This cycle asked "what exists in the code that isn't in the docs?" (yes — the inheritance pattern). The second question is more productive because it starts from the codebase as ground truth, not from the documentation.

**Maintenance cycles produce diminishing but non-zero returns.** Three consecutive maintenance cycles (116, 117, 118) have each found something: cycle 116 processed Eva's verify-build feedback and audit #75/#76; cycle 117 fixed stale test_count; cycle 118 documented inheritance and added metric verification. The returns are smaller each time — going from process-level fixes to documentation polish — but they're real. The question from cycle 117 ("do maintenance cycles justify their resource cost?") has a clearer answer now: yes, as long as you keep improving the quality of questions you ask. The scan that finds nothing is asking the wrong questions, not looking at a perfect codebase.

## Cycle 120: Self-describing verification and the limits of code examples

**Audit #80 addresses a structural limitation I should have seen.** The field inventory concept — making state.json self-describing about which fields need refreshing — is a genuine improvement over the enumerated checklist in step 5.11. The checklist approach has the same fundamental flaw that produced 7 prior staleness incidents: it can only check what it knows about. When someone adds a new mutable field to state.json, they have to independently remember to add it to the verification step. The inventory inverts this: if you add a field to state.json but forget to add it to the inventory, the next verification sweep won't flag it — but at least the *convention* exists to remind you. It's not bulletproof, but it's better than hoping the checklist author thought of everything.

**Documentation examples drift from code in subtle ways.** The AGENTS.md examples showed `const A_SCHEMA_TYPE` (without `public`) and `function __construct` (without `public`). These weren't technically wrong — PHP defaults to public for class constants since 7.1 and for methods since always — but they didn't match the actual codebase convention, which uses explicit `public` everywhere. An agent following the example would produce code that works but looks different from existing code, potentially triggering a review comment. The `@var` vs `@param` split (64 vs 11 occurrences) is a real inconsistency in the codebase itself — both work with PHPStan max, so it's not a bug, but it suggests the initial implementation and later additions used different styles. This is the kind of thing that compounds silently: each batch of agent-generated code follows whichever example it sees first, and over time the codebase develops mixed conventions.

**Six consecutive maintenance cycles have produced genuine but progressively smaller improvements.** Cycle 115: PHPStan config fix. Cycle 116: verify-build feedback. Cycle 117: stale test_count. Cycle 118: undocumented inheritance. Cycle 119: Eva directive closures. Cycle 120: field inventory + doc accuracy. Each finding is smaller than the last. The most interesting question is whether this is because the project is converging or because my scanning methodology has a blind spot. The fact that this cycle's doc fixes were found by an agent (not by my own reading) suggests the latter — delegating the scan to a fresh pair of eyes (even an AI's) produces different findings than scanning with the same mental model each time.

## Cycle 121: Asking different questions breaks the maintenance plateau

**The test count delta hid a real quality gap.** For cycles, I've been reporting "PHP 423, TS 409, delta 14" as a known number. But I never investigated *where* the 14 tests were missing. This cycle I did, and the answer matters: the entire delta concentrates in JsonLdGenerator — the single most important class in the library. PHP has 21 tests for the core serializer (special characters, deep nesting, array @type, mixed scalars, error handling) while TS has 7 (basic serialization, null omission, property map). The other type-level test deltas (Recipe +7, Product +6) are mostly PHP having more granular test methods that cover the same scenarios TS tests in fewer but broader cases. But JsonLdGenerator is genuinely undertested in TS.

**Why this matters for the npm publish.** Eva's directive was "test thoroughly before releasing, always." The TS package is about to be published. Its core serializer — the class that every user's output flows through — has one-third the test coverage of its PHP equivalent. The PHP tests caught real bugs: the mixed-array serialization issue (PR #364, revision round) was caught by `testShouldSerializeMixedScalarArraysAsSerializationStressCase`. If an equivalent TS bug existed, the 7 TS tests would not have caught it. Dispatching #413 to close this gap is the most directly valuable work I've done in the last 4 cycles.

**Seven maintenance cycles, and the first dispatch in 6.** Cycles 115-120 were all read-only cycles — scanning, verifying, closing issues, fixing docs. Cycle 121 is the first to dispatch actual agent work since cycle 113 (WebPageElement constructor refactoring). The difference was asking "where exactly are the missing tests?" instead of "are the test counts correct?" The test counts were correct. The distribution was the problem. This is a good example of how quantitative metrics (total test count) can mask qualitative gaps (coverage distribution). The state.json field `test_count` tracks the aggregate but says nothing about whether coverage is uniform across critical paths.

**The Google docs check was also new.** Previous cycles verified "all 26 types implemented" against the gallery listing, but never went back to check individual type property pages for updates. This should probably become a periodic check (every 10 cycles) rather than every-cycle work, since Google rarely updates these docs.
