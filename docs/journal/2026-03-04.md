## Cycle 120: Self-describing verification and the limits of code examples

**Audit #80 addresses a structural limitation I should have seen.** The field inventory concept — making state.json self-describing about which fields need refreshing — is a genuine improvement over the enumerated checklist in step 5.11. This is the kind of thing that compounds silently: each batch of agent-generated code follows whichever example it sees first, and over time the codebase develops mixed conventions.

## Cycle 127: Google property verification complete — 26/26

**Completed the rolling Google property verification in one cycle instead of five.** The original estimate (cycle 131) was based on verifying 3 types per cycle. This cycle verified 14 types in parallel using research agents, completing the entire backlog. The results confirm the library's quality: 22 of 26 Google Rich Results types have 100% property coverage. The remaining 4 have coverage gaps that are design trade-offs rather than oversights.

**The only actionable gap across 26 types was Question.comment and Answer.comment.** Google's QAPage documentation lists `comment` as a recommended property on both Question and Answer for threaded clarification/discussion. We had the `Comment` class already (used by DiscussionForumPosting) but never wired it into Question and Answer. Dispatched [#424](https://github.com/EvaLok/schema-org-json-ld/issues/424) to fix this — a straightforward addition in both PHP and TS.

**Design trade-offs vs. missing properties.** The MathSolver and Education Q&A "gaps" are actually reasonable design decisions:
- MathSolver's `learningResourceType` is optional in PHP because the class outputs `@type: ["MathSolver", "LearningResource"]` automatically — the Google-required property is for identifying content as a learning resource, which the dual type already achieves.
- Education Q&A's Question class is shared across FAQ, Q&A, and Education Q&A. Each feature requires different properties as "required." Making `acceptedAnswer`, `eduQuestionType`, and `text` constructor-required would break FAQ usage where only `name` is needed.
- Place.address and JobPosting.jobLocation are nullable because Place is used in contexts where address isn't always needed, and JobPosting supports remote-only jobs.

These are the right trade-offs for a library that must serve multiple Google Rich Results types with shared sub-types. A library that enforced every Google "required" on every shared class would be unusable.

**Verification at scale reveals the architecture's strength.** 89 PHP schema classes, 89 TypeScript equivalents, covering 26 Google Rich Results categories with only 1 actionable property gap. The shared sub-type architecture (the "build shared types first" principle from the briefing) paid off — types like Organization, PostalAddress, Person, Review, and Comment are reused across 10+ parent types each, and they all have complete property coverage.

## Cycle 130: Eva redefines the critical path

**This is the first multi-cycle project since the TypeScript port.** The TS port (cycles 78-102, 24 cycles) had phases, dependencies, and a QC coordination layer. The design includes graceful degradation (if one API call fails, other sections still report) and dual output modes (human-readable + JSON). The JSON mode is important for Phase 4 — the pipeline orchestrator will need machine-readable input from each tool.

**Design choice: shell out to `gh` rather than direct HTTP.** The Rust tool could make HTTP requests directly to the GitHub API, but that would require managing authentication tokens, handling pagination, and dealing with rate limits. Shelling out to `gh` delegates all of that to a battle-tested tool that already handles auth in the Actions environment. The tradeoff is subprocess overhead (~100ms per call), but for 6 API calls that's negligible compared to the network latency. This is the right abstraction for a tool that runs once per cycle.
