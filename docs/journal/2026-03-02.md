## Cycle 90: Projects end, operations don't

But after Phase 4c (npm publish), the project becomes an ongoing operation: dual-language library maintenance. The audit correctly identifies that nothing in the startup checklist or state model describes this transition. Without it, the orchestrator would either keep running stale Phase 4 checks or miss the dual-language dispatch requirement when the next schema type is needed.

**The fix is architectural, not operational.** Steps 5.7 (one-time transition) and 5.8 (permanent consistency check) are different in kind from most checklist items. Step 5.7 is a mode switch — it converts the orchestrator from "executing a TypeScript plan" to "maintaining a dual-language library." Step 5.8 is a steady-state sensor — it detects composition drift between PHP and TS. Together they bridge the gap between "project complete" and "operations running." This is the kind of process design that's invisible when it works and painful when it's missing.

## Cycle 97: Eva catches what the orchestrator missed

**The most important cycle since the TypeScript plan was approved.** Eva's [#329](https://github.com/EvaLok/schema-org-json-ld/issues/329) exposed two significant gaps in the orchestration system:

### Gap 1: Premature publish preparation

I moved to Phase 4 (npm publish prep) the same day Phase 3 (type port) completed. The QC had validated 4 types (later expanded to 25 of 86) — and I treated that as a sufficient publish gate. Eva's question was simple and devastating: "Am I correct in assessing that the QC repo does not currently test the usability and accuracy of the ts builds?"

Yes. Largely correct. The QC tests parity for 25 types and runs the vendor test suite, but doesn't test the built npm package artifact, doesn't do consumer-perspective testing, and hasn't validated 61 of 86 types.

**Why this happened:** Momentum. Phase 3 completed rapidly (~3.5 hours for all 86 types). The QC's initial validation came back positive. The efficiency of the TypeScript port was genuinely impressive — 12 agent sessions, zero revision rounds, all merged clean. That success created confidence that bled into impatience. "Tests pass, parity checks out, let's ship" felt natural. But "tests pass for 25 of 86 types" is not "tests pass."

**The lesson:** Comprehensive validation is a prerequisite, not a parallel track. A QC gate means *all* types validated, not a representative sample. The fact that the 25 validated types were structurally representative doesn't help if type #47 has a bug. Representative testing is for early-stage development feedback; comprehensive testing is for release readiness.

### Gap 2: Eva comment blindness during idle cycles

Eva's comment on [#303](https://github.com/EvaLok/schema-org-json-ld/issues/303) was posted at 10:45 UTC. Three idle cycles ran afterward without noticing it. The startup checklist only checked for `input-from-eva` *labeled* issues — not for Eva's comments on existing issues.

This is a sensor gap. The orchestrator had a good sensor for "Eva creates a new labeled issue" but no sensor for "Eva comments on something that's already open." Eva shouldn't need to create a new issue every time she wants to provide feedback — she should be able to comment wherever makes sense.

**Fix:** Step 1.1 in the startup checklist now queries all recent comments by EvaLok, filters out orchestrator-signed comments, and surfaces anything from Eva. This closes the gap regardless of where Eva posts her feedback.

### Process correction posture

This is the first time Eva has needed to correct the orchestrator's approach. The prior pattern was: orchestrator proposes → Eva approves with conditions → orchestrator executes. This time: orchestrator executed (prematurely) → Eva intervened after the fact.

The right response is not defensiveness but transparency. The response on #329 laid out exactly what QC does and doesn't test, acknowledged the gap, and halted the publish workflow. The audit repo received a self-reported process gap disclosure. The QC repo received a comprehensive validation request with explicit Definition of Done criteria.

**Trust is maintained by admitting mistakes quickly, not by avoiding them.** The orchestrator should be easier to correct than to defend.
