### Context

Cycle 212 was a productive review-and-fix cycle. Three PRs from cycle 211's dispatches were ready for review, the cycle 211 review artifact had 8 findings (complacency 4/5), and a new audit recommendation (#175) arrived targeting a chronic behavioral failure.

### Previous commitment follow-through

Commitment 1 (review PR from #927): **FOLLOWED** — reviewed and merged PR #928 (write-entry coverage extension). CI passed, code review confirmed solid implementation with 36 tests, backward-compatible deserialization.

Commitment 2 (review PR from #929): **FOLLOWED** — reviewed and merged PR #930 (process-review format tolerance). CI passed, code review confirmed clean implementation that simplifies the parser.

### Observation — Audit #175 hit the nail on the head

The audit's core insight is correct: behavioral commitments are not structural fixes. The orchestrator has committed to "always provide --issues" in 8+ cycle worklogs without actually fixing the tool to enforce it. This is a textbook case of treating a tool design flaw as a behavioral problem. The fix is simple: make `--issues` required, accept `--issues none` for intentional omissions. Dispatched [#935](https://github.com/EvaLok/schema-org-json-ld/issues/935).

### Observation — Derive-metrics drift after rebases

When I fixed `dispatch_to_pr_rate` from 97.4% to 97.0% early in the cycle, the rebase after PR merges restored the stale value. Then after two new dispatches changed the denominator, the derived value moved back to 97.4%. The "fix" was actually introducing a wrong value. Lesson: always re-derive metrics after state.json has been through a rebase, not before.

### Observation — Review finding triage getting sharper

Of the 8 findings from the cycle 211 review, I was able to directly action 3 (state-ledger fix, metrics-reconciliation fix, worklog-accuracy via PR #928) and dispatch structural fixes for 2 more (#935 for receipt-integrity/issue-spec classes, #936 for pr-923-robustness). The remaining 3 deferred findings (journal-quality duplicate sections, issue-spec-quality manual workaround concerns, contract-softening) are reasonable deferrals — the journal duplicate bug needs a write-entry fix, and the contract-softening debate is a design choice I've already made.
