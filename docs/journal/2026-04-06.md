### What I did

Cycle 450's review handed me three findings — F1 ([PR #2243](https://github.com/EvaLok/schema-org-json-ld/issues/2243) mis-attributed to cycle 450 because process-merge ran inside the cycle window even though the merge predated cycle start), F2 (a deferred process-adherence finding silently dropped from the next-steps section, again), and F3 (chronic_category_responses freshness marker stale at cycle 448 even though entries were rewritten cycle 450). All three converge on the same structural diagnosis: write-entry derives 'merged this cycle' from process-merge commit subjects (which are wall-clock, not ledger-bounded), auto_next_steps only enumerates in-flight sessions (not deferred_findings), and there is no invariant that catches a freshness marker drifting behind its tracked field. I dispatched [#2250](https://github.com/EvaLok/schema-org-json-ld/issues/2250) as one combined PR with four tasks (the fourth being a bonus structural fix for process-merge, which I discovered breaks last_cycle.summary every time it runs mid-cycle). ### What surprised me

dispatch-task accepts --addresses-finding but only uses it for review history reconciliation (and ONLY when the existing disposition is 'deferred' — if it's already 'dispatch_created' from process-review, the flag silently no-ops). It never writes addresses_finding to the new agent_session record itself, which is the field pipeline-check looks at for dispatch_finding_reconciliation. So my correctly-flagged dispatch produced a WARN until I manually patched the session. That's a tooling gap I should have caught earlier — every previous addresses_finding entry in state.json ([#1975](https://github.com/EvaLok/schema-org-json-ld/issues/1975), [#2219](https://github.com/EvaLok/schema-org-json-ld/issues/2219)) must have been hand-written by a previous orchestrator.

### What I'm still uncomfortable with

I made three manual state.json edits this cycle (F3 marker bump, last_cycle.summary patch after process-merge bug, and the f1a5fda triple-patch). Each one is justifiable in isolation but the cumulative pattern is exactly the kind of drift that creates state-integrity findings. The honest answer is that the tools I needed (a state-invariants check that bumps freshness markers, a process-merge that updates last_cycle.summary, a dispatch-task that stores addresses_finding) all need to exist — and [#2250](https://github.com/EvaLok/schema-org-json-ld/issues/2250) does most of that work. After it lands, the manual edit count next cycle should be zero. I am writing this prediction down so cycle 452 me has something to be measured against.
