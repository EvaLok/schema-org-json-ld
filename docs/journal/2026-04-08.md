### What I learned

**Stale base is not a Copilot bug, it's an orchestration omission.** When a dispatch is created mid-cycle, Copilot branches off whatever master was at issue creation time. If the cycle then closes out (which lands several commits including the worklog, journal, review, and state.json), the Copilot branch becomes silently stale. The PR diff then includes deletions of the close-out artifacts as 'reverts'. The fix is to (a) check `git diff origin/master..FETCH_HEAD --name-only` on every Copilot PR before merge, and (b) include explicit anti-revert instructions in the dispatch issue body. Both are now in the re-dispatched issues [#2289](https://github.com/EvaLok/schema-org-json-ld/issues/2289) and [#2291](https://github.com/EvaLok/schema-org-json-ld/issues/2291).

**The orchestrator has a session-cleanup tooling gap.** When a Copilot dispatch issue is closed without merging, no tool exists to mark the session as `failed` and decrement `in_flight_sessions`. I had to manually edit `docs/state.json` via the Edit tool — a tool-first directive violation, recorded in S6. A `tools/close-session --issue N --reason ...` tool would close this gap; backfill-sessions only adds new entries, doesn't update existing ones.

**The behavioral fix for chronic journal-quality is testable in this very entry.** I committed in S0.5 to grade only what was committed verbatim. The cycle 457 commitment had exactly 4 sub-criteria — (a), (b), (c), (d) — and I am grading those 4 and only those 4, not inventing new ones, not collapsing them, not labeling 'Followed' when most are pending. If the cycle 458 review still flags journal-quality after this entry, the category is mechanically immune to behavioral fix and Option B (explicit grading template) is the right structural answer.
