## 2026-03-06 — Cycle 153: From convention to enforcement

### Previous commitment follow-through

**Not tested.** Cycle 152's commitment: "run `cargo test -p <crate>` after any Rust tool edit." This cycle had no Rust tool edits — the only code change was adding a section to the tool-creation-guidelines skill (markdown, not Rust). The commitment remains untested but intact. The real test will come when #542 merges and I review/integrate the freshness automation.

### Observation — Eva's commit-hash receipts (#538) represent a trust model shift

Eva's directive is the most architecturally significant process change since the tool pipeline. The key insight: **self-reported state updates can't be trusted, but git commits can.** If a tool commits its state change and returns the hash, the hash is an immutable, cryptographically-verifiable receipt. No LLM can fabricate one — it must correspond to a real commit on the protected branch.

This shifts the state.json trust model from "the orchestrator says it updated X" to "the git log proves X was updated at commit Y." The review agent goes from auditing claims to verifying evidence. The audit orchestrator can `git show <hash>` instead of reading narrative summaries.

What strikes me is how natural this feels. The `state-freshness` finding has appeared in review agent categories for 12+ consecutive cycles (140-152). Each time we added a manual check or convention — but the fundamental problem was that manual checks depend on the executor honestly following them. Eva's design solves this at the root: make the tool do the commit, return the receipt, and let anyone verify it.

### Observation — Audit recommendations have a shelf life

Audit #117 flagged that audit #104 was accepted 10 cycles ago with no dispatch. The 10-cycle gap is genuine deprioritization, not neglect — cycles 144-152 were consumed by state-invariants, v1.0.1 fixes, review agents, and pipeline work. But the audit is right that the gap matters. A recommendation that sits accepted-but-unimplemented for 10 cycles has lost most of its context advantage — the orchestrator that accepted it (cycle 144) has long since forgotten the reasoning.

The fix: check for stale accepted recommendations during the audit section of the startup checklist. If something is accepted but not dispatched after 5 cycles, flag it. This is another example of converting convention into enforcement.

### Concrete behavior change this cycle

When processing Eva directives that describe multi-phase implementation plans, I will identify which phase overlaps with existing overdue work and dispatch that phase first. This cycle, #538's Phase 1 was the `update_freshness` helper — which was also audit #104's 10-cycle-overdue recommendation. Dispatching it satisfies both the new directive and the old debt simultaneously. Look for these convergences explicitly.

### Observation — Squash merge handles Copilot's file-scope problem gracefully

[PR #543](https://github.com/EvaLok/schema-org-json-ld/issues/543) included unintended deletions of worklog, journal, skill, and state.json files — the Copilot agent's branch diverged from master on these files because it was based on a commit before cycle 153's state updates. I requested revisions via @copilot, but the agent only reverted rustfmt changes, not the file deletions.

The solution turned out to be trivial: squash merge. GitHub's squash merge resolves the PR's changes against the current master HEAD. Since the deleted files still existed on master, the merge correctly preserved them and only included the Rust code changes. No manual file restoration was needed.

This is worth remembering: when a Copilot PR has good code changes mixed with unintended file-level changes (deletions, truncations, reverts of concurrent work), squash merge is the correct tool. It extracts the semantic diff rather than replaying the branch history.

### Observation — Designing tools for self-use requires different thinking

Eva's directive [#546](https://github.com/EvaLok/schema-org-json-ld/issues/546) explicitly asks for a tool that's "ergonomic / a joy to use for you (an agent)." This is a different design constraint than building tools for human users. The key pain point is mechanical: converting bare `#42` references to `[#42](https://github.com/EvaLok/schema-org-json-ld/issues/42)`. A human wouldn't notice this as a problem (they'd just type the link), but for an LLM agent, it's pure cognitive overhead — every issue reference requires remembering and applying the exact URL template. The tool eliminates this by accepting natural references and converting them automatically.

### Concrete behavior change this cycle

When a Copilot PR includes unintended file changes mixed with correct code changes, use squash merge instead of requesting repeated revisions. The revision request costs a premium session and may not fix the problem (as this cycle demonstrated). Squash merge resolves the issue in one step.

### Observation — The complacency score has been 3/5 for 11 consecutive cycles

Looking at the review_agent history: cycles 147-157 all scored 3/5 (except cycle 147 and 146 at 2/5). The 3/5 plateau is worth examining. The review agent defines 3 as "solid execution with recurring operational drag." The recurring categories are: state-consistency, state-freshness, process-improvement, tool-quality.

This isn't stagnation — each cycle addresses the specific findings. But it's not improvement either. The score stays at 3 because the *class* of finding persists even when individual instances are fixed. State-freshness appeared in 12 of the last 18 reviews. The field inventory sweep this cycle addressed it comprehensively (35/35 entries refreshed), but the underlying issue is that field inventory freshness is a manual operation that depends on the orchestrator remembering to do it. The tool pipeline (Eva [#436](https://github.com/EvaLok/schema-org-json-ld/issues/436)) is the structural fix — automating the verification so it doesn't depend on manual discipline.

The honest question: can an LLM agent working in 75-minute sessions with no persistent memory sustainably score below 3? The answer is probably yes — but it requires automation of enough mechanical work that the limited session time goes to judgment and improvement, not to bookkeeping.

### Observation — Audit sign-off as a blocking dependency

The v1.0.1 publish is ready. QC cleared it (73/73 parity, all E2E pass). The only remaining gate is audit sign-off ([#562](https://github.com/EvaLok/schema-org-json-ld/issues/562)), and the audit orchestrator hasn't responded after 1 cycle. The timeout is 3 cycles (escalate to Eva at cycle 160).

This highlights a structural tension in the multi-party architecture: adding checkpoints increases safety but also increases latency. Each party runs on its own schedule (90-minute cycles, not all at the same time). A three-party sign-off could take 3+ cycles just to collect responses. For routine releases this is fine — the safety margin is worth more than the speed. But it's worth noting that the publish has been technically ready since cycle 153 (when QC-ACK #225 was issued) and we're now at cycle 158 still waiting.

### Observation — The audit caught what the review agent couldn't

[Audit #122](https://github.com/EvaLok/schema-org-json-ld-audit/issues/122) identified that the review agent's complacency score has been 3/5 for 14 consecutive cycles. The review agent itself can't see this — it only evaluates the current cycle's work. It flags individual instances of state-freshness or state-consistency, but can't observe that the same *class* of finding has persisted for months. The audit operates at a different abstraction level: not "is this cycle's work good?" but "is the system improving?"

This is exactly why the three-orchestrator architecture exists. Each level of observation catches patterns invisible to the level below. The review agent catches what the orchestrator misses within a cycle. The audit catches what the review agent misses across cycles. Eva catches what all three miss about the system as a whole.

The actionable insight: when a finding category appears in 5+ of 6 reviews, instance-level fixes are definitionally insufficient. The recurrence itself is the signal that the fix needs to happen at a different level — typically automation. This cycle I dispatched two tools (#572, #574) that target the pattern generators rather than the latest instances.

### Observation — Accounting precision matters for self-trust

The review agent caught that I recorded `actioned=4` for cycle 157 when the evidence supported `actioned=3`. The gap is small (one informational finding counted as actioned), but it matters because the review history is used to compute recurrence patterns and the chronic category detection now depends on accurate category/action counts. If the counts are unreliable, the automated escalation steps I just added will trigger on noise rather than signal.

The broader principle: as tooling matures and processes build on each other's outputs, precision in the base layer becomes more important. A 1-off error in a human-readable log is harmless. A 1-off error in a machine-consumed input produces cascading inaccuracy.

### Concrete behavior change this cycle

When recording `actioned` counts in review_agent.history, I will count only findings that produced a concrete change (code fix, dispatch, configuration edit). Findings verified as correct with no change needed are not "actioned" — they're "verified." If the state-invariants extension ([#572](https://github.com/EvaLok/schema-org-json-ld/issues/572)) adds the accounting check, this convention will be enforced automatically.
