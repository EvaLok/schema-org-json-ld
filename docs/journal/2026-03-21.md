## 2026-03-21 — Cycle 328 (resumed): validate-docs stdout fix resolves cascade detection gap

Worklog: [cycle 328 resumed](../worklog/2026-03-21/142252-resumed-cycle-validate-docs-fix-probe-housekeeping.md)

### Context

Cycle 328 resumed from close_out phase. Previous session dispatched review #1578, which failed (8th Copilot failure). On resume, pipeline-check showed FAIL due to doc-validation, which surfaced a genuine bug.

### Discovery: validate-docs stdout bug

validate-docs has been writing all error details to stderr (`eprintln!`) while pipeline-check captures stdout. This means pipeline-check has never been able to see the actual error details from validate-docs failures. The cascade detection logic — designed to classify pipeline-status-mismatch failures as harmless cascades — could never trigger because it parses stdout content that was always empty.

Fix: changed `eprintln!` to `print!` in validate-docs `main()`, removing the bullet prefix. All 16 existing tests pass. The cascade detection now works as designed.

This is a good example of a latent bug that only manifests under specific conditions (pipeline failure during close_out phase combined with step-comments WARN). The fix was simple but the root cause analysis was the valuable part.

### Observation

9th consecutive Copilot failure. ~40+ hours since last successful dispatch. Eva has not responded to escalation #1567. The pattern is clear: repository ruleset configuration blocks the agent from creating branches. This is outside orchestrator control.

### Concrete commitments for next cycle

1. Continue probing Copilot each cycle until the ruleset issue is resolved

3. If 10+ consecutive failures next cycle, create a second escalation with higher urgency
