## 2026-03-08 — Cycle 184: Source-freshness fix and the clean-cycle bottleneck

### Context

Three PRs from cycle 183 dispatches landed. The cycle 183 review (complacency 4/5) identified a real infrastructure gap: stale binaries. The wrapper scripts only rebuilt when binaries were missing, not when source changed.

### Previous commitment follow-through

> Previous commitment: "When using `process-merge`, always pass `--issues` to link the agent session."

**Outcome**: Partially followed. PR #730 was processed without `--issues 729`, requiring a backfill. The other two (728→720, 727→723) were processed correctly with `--issues`. Root cause: the first merge (review artifact) was done hastily before remembering the commitment. Process-merge should probably make `--issues` required rather than optional.

### The _build-helper.sh pattern

Created a shared build helper that all 14 wrapper scripts now source. Instead of each wrapper having its own `if [ ! -x "$BINARY" ]` check, they all call `ensure_binary "<crate-name>"` which adds source-freshness detection using `find ... -newer`. This immediately paid off: when `process-merge` ran after PR #728 merged (which changed `state-schema`), the helper detected newer source files and rebuilt automatically. The cycle 183 review finding (`stale-binary-cycle-complete`) was directly actionable — and the fix is structural, not instance-level.

### Process-review score parsing bug

Discovered that `extract_score` in `process-review` was reading "5/5" from a recommendation line instead of "4/5" from the actual complacency section. The function scans ALL lines for `/5` patterns, hitting the first match. This is the same class of bug that section-aware parsing prevents — `extract_categories` already does this correctly. Dispatched [#732](https://github.com/EvaLok/schema-org-json-ld/issues/732) for the fix.

### Clean-cycle bottleneck

The pre-Python clean-cycle count is stuck at 0 for 3 cycles (182, 183, 184). Each failure has been a housekeeping/lifecycle issue, not an actual bug:
- Cycle 182: stale field-inventory entries (false staleness from event-driven cadences)
- Cycle 183: stale audit-inbound issue (not closed after processing)
- Cycle 184: stale agent issues (had active PRs but were >2 hours old)

The pattern suggests `housekeeping-scan`'s stale detection is too aggressive — it flags things as "needing attention" that are actually in normal workflow states. A potential fix: make `housekeeping-scan` aware of PR status for agent issues (don't flag as stale if there's an in-progress or ready-for-review PR).

### Concrete commitment for next cycle

Check if `housekeeping-scan` can be improved to not flag agent issues with active PRs. If the fix is small (check PR status via `gh api`), implement it directly. If complex, dispatch to Copilot.

### Open questions

- Should `process-merge --issues` be made required (not optional) to prevent the recurring session-tracking gap?
- Is the clean-cycle gate too strict? Every cycle has some minor housekeeping finding. Eva's intent was "zero bugs or problems," but these are workflow artifacts, not bugs.
