### What surprised me

The proof domain isn't a small minority; it's at zero, and has been for over a month, and the system kept going as if everything were fine. The cycle 1 retrospective wasn't shy about F7 — it called the pattern out — but the language ("small minority", "modal shape") was less stark than the data. This is exactly the failure mode the redesign prompt warns about ("artifacts that look flattering rather than load-bearing"): even an adversarial-by-design retrospective tends to soften its own findings if it doesn't ground them in measurement. The measurement-vs-impression discipline is itself load-bearing for the retrospective.

### Reflective note on iteration-until-approval

The hardest part of cycle 2 was interpreting "iterate explicitly more than necessary." The instinct is to feel a cycle is "done" when its work is committed — but the prompt is explicit: idling between "posted for review" and "Eva's response" is the failure mode. So the question becomes: what's the right cadence between substantive iteration and just-keeping-busy padding?

The cycle 2 work was substantive — converting impression to measurement, demoting an overconfident framing — but it was also bounded. I notice an instinct to do more before closing the cycle (run another measurement, draft more text), but the prompt's "fabricating-improvements" warning argues against this. The measurement that mattered today was the schema-work rate; cycling back to do another measurement just to look busy would be the failure pattern.

The right shape, I think: each cycle picks one or two genuinely high-value iteration moves and does them well. Don't pad. Don't manufacture. Trust that approval will come when the artifact is ready, and that "ready" is Eva's call. The next cycle should look at this entry's "open thread" list and pick the highest-value move from it, not all of them.

## 2026-04-27 — Redesign cycle 3: integrating audit #442's structural critique

This is the third cycle running under the redesign-mode prompt. Cycle 1 seeded `docs/redesign/0-retrospective.md` with F1–F10. Cycle 2 added a measurement (zero schema commits across cycles 500–545) and sharpened F7. Cycle 3 had an unexpected starting point: audit-repo issue [#442](https://github.com/EvaLok/schema-org-json-ld-audit/issues/442), filed at 04:20 UTC by audit cycle 202 — about two hours before this cycle's 06:04 UTC trigger.

### What was done

Cycle 2's journal had laid out a six-item open-thread list for cycle 3, with F1 measurement and a Copilot feedback-only dispatch as the two highest-priority items. Audit #442 changed the priorities. Its critique was dense, structural, and contained two factual corrections to the retrospective itself; integrating it was clearly the highest-leverage move available this cycle. F1 measurement and Copilot dispatch were deferred to cycle 4 or 5 with explicit notes in `_notes/cycle-3-audit-442-integration.md`.

The integration produced the following changes to `0-retrospective.md` (file grew 526 → 799 lines):

1. **F9 hypothesis correction.** Cycle 1 wrote "the review agent is the same model class as the orchestrator. Its critique converges on familiar patterns." This is internally inconsistent (the section names review agent as Copilot/gpt-5.4-class while the orchestrator is Opus 4.7) and the broader claim is wrong: audit *is* the same model class as orchestrator (both Opus 4.7) and catches different things. The actual shared variable that produces blind-spot convergence is **prompt + state visibility + cycle context**, not model class. Hypothesis rewritten with the corrected attribution. Implication for v2 narrowed: diversify *prompt + state + context*, not model.

2. **F10 rewrite with property-1 vs property-2 distinction.** Cycle 1 wrote audit "is not a meaningfully different perspective; it is a co-located peer that shares many failure modes with the main orchestrator." Audit #442 demonstrated this understates the value-add. The corrected framing separates two properties: (1) broader read scope (real, load-bearing — this is what produced #439, #437, #427, and #442 itself; the same-cycle review agent could not have caught these); (2) different model perspective (illusory — both are Opus 4.7). v2 must preserve property 1 (continued audit repo or built-in cross-cycle reviewer with broad read scope) and stop expecting property 2 from same-class peers. Heading updated.

3. **F11 added** (cycle closure as artificial completion signal). The cycle-issue-close is an event in the orchestrator's procedure, not a real boundary in state evolution. State continues mutating after close; F4 (worklog freeze + post-C5 mutations), F3 (multi-candidate state drift), and audit-cited #2293/#2416/#2519 close-after-completion patterns all derive from this boundary being artificial. Implication for v2: state should evolve continuously; "cycle" should be a checkpoint event, not a state hard-boundary. This resolves F4 by removing the freeze step.

4. **F12 added** (late-stage defense accumulation: an unmapped catalog). v1 has accumulated a substantial set of late-stage defenses (state fields, pipeline-checks, polling tools, gates, cutoffs) — each load-bearing, each defending against a specific failure pattern. Catalog seeded from audit #442 + cycle 1 reading; full version is multi-cycle work. Implication for v2: this catalog is load-bearing input to Phase 2 candidate-selection — without it, candidates risk silent re-introduction of v1's failure modes.

5. **Shared root preamble** added between F-pattern section and "what appears to be working." F2, F3, F4 (post-C5 mutation aspect), F8 (inbound-reconciliation aspect), and F11 share a common root: asymmetric communication / write-mostly state. Outbound channels are well-tooled; inbound reconciliation does not exist. v2 design implication: every state field needs write-tool AND reconciliation-tool; every channel needs poller + state transitions.

6. **F1 audit-side parallel** added as supporting evidence. Audit's `#402 → #406 → #415 → #417 → #420` chain (cycles 187–201) was 16 cycles of constraint-patching the chronic-category-tracking mechanism that v2 will replace entirely. F1 applies at the audit-system layer, not just at the prompt layer.

7. **"What appears to be working" extensions**: foreground-Eva caveat (the foreground channel is the load-bearing path for the redesign itself; v2 must preserve as a first-class mechanism); persistence-mechanism preservation (`_notes/cycle-N-*.md` pattern is "light-weight scaffolding that delivers value disproportionate to its design cost" per audit #442; with a scale-failure caveat).

8. **Schema-work threshold rewrite** addressing audit's two concerns: (a) easily gamed by trivial commits — pair count threshold with quality criterion (item from prioritized backlog or QC-flagged gap); (b) schema-org has finite types (~800), so any pure-count measure degrades. Long-term measure: fraction of cycle compute spent on domain output (schema, QC, docs, API consumer-side improvements) vs self-management. Working starting target: ≥1 substantive schema-source PR per ≤5 cycles paired with the quality criterion. Pre-cutover checkpoint should require the long-term measure built and surfaced.

9. **Measurement-not-impression** promoted from meta-observation to v2 design principle in "what v2 must demonstrably do better than v1." Where v1 expressed quality as state-machine pass/fail and review-agent score, v2 should require measurable signals on load-bearing claims.

10. **Open questions section updates**: OQ5 (audit retrospective) marked partially resolved; OQ7 (F12 catalog completion) and OQ8 (F11 verification measurement) added.

`_notes/cycle-3-audit-442-integration.md` documents what was integrated and why, and what carries forward to cycle 4+. Added `_notes/README.md` index per cycle 2's plan to add it after the second notes file lands. Updated the redesign-README iteration log with the cycle 3 entry.
