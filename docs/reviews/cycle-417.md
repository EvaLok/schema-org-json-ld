## 1. [worklog-accuracy] The write-entry patch pipeline still preserves stale primary next steps after promoting the worklog to its final post-dispatch form

**File**: tools/rust/crates/write-entry/src/main.rs:512-518,729-760,7366-7391
**Evidence**: PR #2054 fixed the primary in-flight counter, but `execute_patch_pipeline()` still sends populated `next_steps` through `patch_or_addendum_numbered_section()`. That helper preserves any non-placeholder `## Next steps` block and appends `## Next steps (post-dispatch)` instead of replacing it. The regression test `patch_pipeline_preserves_sections_after_next_steps` locks that behavior in. Cycle 417 followed that exact path. Frozen commit `2b6ef69` recorded `- **In-flight agent sessions**: 0` with `1. No in-flight sessions ‚Äî plan next dispatch`. Refresh commit `43b9ae6` changed the same worklog to `- **In-flight agent sessions**: 1` and added a post-dispatch review step for `#2058`, but it left the old `No in-flight sessions` line intact. The final artifact therefore contains mutually exclusive next-step narratives.
**Recommendation**: When refreshing a finalized worklog after C6, either replace the primary `## Next steps` block with current state-derived steps or keep the original block explicitly labeled as a pre-dispatch snapshot with its disclaimer.
