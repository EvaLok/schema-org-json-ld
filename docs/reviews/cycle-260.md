## 1. [code-quality] The post-step hotfix still fails open on real paginated `gh api` output

**File**: /home/runner/work/schema-org-json-ld/schema-org-json-ld/tools/rust/crates/post-step/src/main.rs:227-236
**Evidence**: Direct push `b82a376` removed the unsupported `--slurp --jq` combination, but the replacement parser only repairs the exact string `"][` by doing `raw.replace(\"][\", \",\")` and then `unwrap_or_default()` on any remaining parse error. GitHub CLI documents `gh api --paginate` as emitting one JSON document per page, separated by newlines rather than merged into a single array, so a normal multi-page response like `[{...}]\n[{...}]` still fails both parse attempts. In that case `existing_comments()` returns `Ok(vec![])`, which silently disables duplicate-step detection instead of failing closed. The tests in this file only exercise mocked `existing_comments()` values and never cover runner parsing of multi-page `gh api` output, so this regression path remained invisible.
**Recommendation**: Parse paginated output as a stream of JSON values or explicitly split pages on JSON boundaries/newlines, and return an error if any page cannot be parsed. Add a regression test that feeds multi-page `gh api --paginate` output into the real parsing path.
