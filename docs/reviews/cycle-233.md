## 3. [tooling] `cycle-receipts` still uses a mutable `last_cycle.timestamp` as the current-cycle start, so close-out queries drop most of cycle 233

**File**: tools/rust/crates/cycle-receipts/src/main.rs:149-172
**Evidence**: For the “current” cycle, `resolve_cycle_window()` sets the receipt window start to `docs/state.json last_cycle.timestamp` instead of a stable cycle-start boundary. That field is mutated during the same cycle: at cycle start (`git show 47bf641:docs/state.json | jq '.last_cycle, .cycle_phase'`) it is `2026-03-12T06:16:21Z`, but after close-out (`git show f2fe1cb:docs/state.json | jq '.last_cycle, .cycle_phase'`) it becomes `2026-03-12T06:34:31Z`. Because `current_cycle_from_state()` reads `last_cycle.number`, querying cycle 233 after close-out treats 233 as the current cycle and uses the later timestamp as the window start; that is why canonical receipt discovery now returns only `f2fe1cb`, `5257359`, and `11312ee` and drops earlier cycle-233 receipts like `6d99476`, `773ac05`, `be83c5b`, and `9c0971a`. The bundled test `collect_receipts_uses_state_timestamp_for_current_cycle()` (`tools/rust/crates/cycle-receipts/src/main.rs:325-345`) locks this behavior in.
**Recommendation**: Derive the current-cycle window from an immutable cycle-start marker (or the cycle-start commit) instead of `last_cycle.timestamp`, and add a regression test that queries receipts after `cycle-complete` to prove earlier same-cycle receipts remain visible.
