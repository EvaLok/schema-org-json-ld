## 1. [tool-correctness] Status-aware dispatch logic still misclassifies the repository's `closed` terminal status

**File**: tools/rust/crates/verify-review-events/src/main.rs:344-357
**Evidence**: `status_expects_merge_outcome()` only treats `failed`, `closed_without_pr`, and `closed_without_merge` as terminal non-merge statuses. In this repository's live status taxonomy, `closed` is also terminal: `record-dispatch` handles `Some("closed_without_merge") | Some("closed") => {}` so those sessions do **not** imply a merge outcome (`tools/rust/crates/record-dispatch/src/lib.rs:122-128`), and `state-invariants` likewise counts `Some("closed_without_merge") | Some("closed")` in the terminal closed-without-merge bucket (`tools/rust/crates/state-invariants/src/main.rs:1233-1245`). Because unknown statuses default to `true`, `verify-review-events` currently treats a `closed` session as if a merged PR should exist. The new tests never cover `closed`, even though `docs/state.json` still contains `status: "closed"` sessions.
**Recommendation**: Add `closed` to the terminal non-merge branch, then add regression tests covering both `closed` and the intentionally special-case `reviewed_awaiting_eva` status so the verifier matches the repository's real status model instead of a partial recollection of it.

## 2. [test-quality] The new cycle-receipts priority tests lean on unreachable synthetic inputs instead of the real parser path

**File**: tools/rust/crates/cycle-receipts/src/main.rs:163-171,545-575
**Evidence**: `extract_match_steps()` can emit at most two labels for a real commit subject: it pushes the single result of `extract_step(subject)` and, separately, `cycle-tagged` when `[cycle N]` is present. But the headline `deduplication_handles_triple_match()` test feeds `deduplicate_receipts()` three labels for the same SHA, including `review-history`, even though the supplied commit subject is `state(process-review): reviewed change [cycle 198]` and `extract_step()` can only return `process-review` for that subject. The parser can never emit `review-history` from that input. That means the advertised 3-tier priority system is being validated through hand-constructed impossible `ReceiptMatch` vectors rather than through `collect_receipts()` on real parsed commits. There is still no end-to-end test proving that a real `state(process-merge)... [cycle N]` commit becomes one canonical row plus an alias after parsing.
**Recommendation**: Add a TempRepo-backed test that exercises `collect_receipts()` or `render_markdown()` with real commit subjects that produce duplicate SHA matches through the parser, and trim or demote the impossible triple-match case so the test suite documents reachable behavior instead of imaginary inputs.

## 3. [infrastructure-consistency] Audit #259's two-state chronic gate was documented, but not structurally enforced

**File**: tools/rust/crates/state-invariants/src/main.rs:950-964
**Evidence**: Cycle 271 added startup checklist step 0.5.12 requiring two distinct chronic verification states: string-valued `tool_hardened` / pending runtime proof, and numeric `runtime_verified`, with only the latter clearing the chronic category (`STARTUP_CHECKLIST.md:97-103`). But the invariant that guards chronic verification does not distinguish those states. It simply checks whether `verification_cycle` is non-null and passes immediately. The corresponding test only covers a numeric value (`tools/rust/crates/state-invariants/src/main.rs:1986-2029`). In other words, the repository now records `verification_cycle: "270-tool-hardened, pending-code-PR-runtime-proof"` for `state-integrity` and `review-evidence`, but the invariant layer treats that exactly the same as a real runtime-verified numeric cycle.
**Recommendation**: Update `state-invariants` so string-valued intermediate states remain visibly open (or at least separately classified) until a numeric runtime verification cycle is recorded, and add tests covering both the new string form and the final numeric form. Otherwise the "two-state gate" is just prose, not an enforced guardrail.

## 4. [state-integrity] Cycle 271 rewrote chronic responses without refreshing the corresponding field-inventory marker

**File**: docs/state.json:4007-4010,4192-4193,4217-4218
**Evidence**: Both chronic entries touched this cycle now carry `updated_cycle: 271` and the new string-form `verification_cycle` values for `state-integrity` and `review-evidence`. But the field inventory entry for `review_agent.chronic_category_responses` still says `last_refreshed: "cycle 270"`. **Recommendation**: Refresh `field_inventory.fields["review_agent.chronic_category_responses"].last_refreshed` whenever those entries are edited, and add or extend a state invariant so chronic-response updates cannot land without the matching freshness marker moving in the same cycle.
