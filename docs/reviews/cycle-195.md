# Cycle 195 Review

I reviewed cycle 195 against the merged code, `docs/state.json`, the cycle worklog/journal, and the current write-side/tooling behavior. The cycle did repair the cycle-194 review-history entry and the clean-cycle counter, but the deeper pattern is still narrative overreach: the new pipeline phase does not catch the live metric drift it claims to cover, the ownership-consolidation story is still broader than the code supports, and the repeated `disposition-overstatement` deferral was justified as compliance with a checklist step it did not actually satisfy.

## Findings

1. **[pipeline-integrity] The new derive-metrics pipeline phase still misses the exact drift it was added to detect**

   The worklog says cycle 195 merged PR #798, closed with `Pipeline status: PASS (6/6 with derive-metrics phase, after invariant fix)`, and reported a `Copilot metrics (from derive-metrics)` block (`docs/worklog/2026-03-08/193800-hundred-ninety-fifth-orchestrator-cycle.md:5-12,22-34`). But the pipeline implementation only compares seven integer counters in `DERIVE_METRICS_FIELDS` (`tools/rust/crates/pipeline-check/src/main.rs:14-22`). Its derive-metrics classifier never checks `pr_merge_rate` or `dispatch_to_pr_rate`; it only walks those integer fields and then reports `tracked copilot_metrics fields match` (`tools/rust/crates/pipeline-check/src/main.rs:340-450`). By contrast, the authoritative `derive-metrics --check` path explicitly validates both rate fields and treats either the percentage display or the legacy `n/d` ratio as acceptable (`tools/rust/crates/derive-metrics/src/main.rs:198-267`).

   The repository state shows the live mismatch. `docs/state.json` still stores `dispatch_to_pr_rate: "218/223"` while the surrounding counters are `total_dispatches: 225`, `resolved: 223`, `in_flight: 2`, `produced_pr: 218` (`docs/state.json:2204-2217`). The same worklog block also says `Resolved: 221`, which is not even arithmetically compatible with the `225 total / 2 in-flight` state it reports (`docs/worklog/2026-03-08/193800-hundred-ninety-fifth-orchestrator-cycle.md:22-28`). When I ran the tools, `bash tools/derive-metrics --check --repo-root /home/runner/work/schema-org-json-ld/schema-org-json-ld` failed with `copilot_metrics.dispatch_to_pr_rate expected 96.9% (or 218/225) but found 218/223`, while `bash tools/pipeline-check --repo-root /home/runner/work/schema-org-json-ld/schema-org-json-ld` still returned `Overall: PASS`.

   **Recommendation:** make the pipeline phase call the actual `derive-metrics --check` logic or extend `collect_derive_metrics_mismatches()` to include `pr_merge_rate` and `dispatch_to_pr_rate`, then add a regression test where integer counters match but a rate string is wrong. Also stop labeling the worklog metrics block “from derive-metrics” unless the final committed state actually passes that check.

2. **[ownership-consolidation-gap] Cycle 195 still overstates metric ownership consolidation**

   The cycle artifacts describe the theme as “completing the copilot_metrics ownership consolidation,” specifically saying `derive-metrics` is now the sole authority for rate fields and narrative summaries, with `record-dispatch` and `process-merge` “only writing raw counters” (`docs/journal/2026-03-08.md:468-486`). The worklog also marks both the drift finding and the ownership-consolidation finding as effectively dispatched/resolved through PRs #796/#798 plus issue #802 (`docs/worklog/2026-03-08/193800-hundred-ninety-fifth-orchestrator-cycle.md:16-20`).

   That is not what the code says today. `record-dispatch` really was narrowed: it reads `total_dispatches`, `in_flight`, and `resolved`, then writes only `total_dispatches`, `in_flight`, `dispatch_log_latest`, and the new `agent_sessions` row (`tools/rust/crates/record-dispatch/src/main.rs:97-183`). A repo-wide search for writes to `/copilot_metrics/(dispatch_to_pr_rate|pr_merge_rate|note)` only turned up `process-merge`. And `process-merge` is still writing all three derive-owned outputs itself: it computes ratio strings for `pr_merge_rate` and `dispatch_to_pr_rate`, rebuilds `copilot_metrics.note`, patches those paths in `docs/state.json`, and refreshes the freshness markers for the rate fields (`tools/rust/crates/process-merge/src/main.rs:152-248`).

   So the remaining overlap is narrower than cycle 194, but it is not “complete ownership consolidation.” It is “record-dispatch stopped being a conflicting writer; process-merge still is.” That distinction matters because the cycle’s own false-green claim depends on exactly this remaining overlap.

   **Recommendation:** narrow the narrative until #802 actually lands. The accurate statement for cycle 195 is that `record-dispatch` was consolidated, and `process-merge` remains the last overlapping writer of derive-owned fields. Do not describe the consolidation as complete while `process-merge` still rewrites the rate fields and note.

3. **[process-adherence] The repeated `disposition-overstatement` deferral is presented as step-0.6 compliance, but it is another deferral of the same escalated item**

   Step 0.6 is explicit: if a review finding has been deferred for 3+ consecutive cycles, it “must be actioned this cycle or explicitly dropped with rationale” (`STARTUP_CHECKLIST.md:63-76`). The history shows `disposition-overstatement` meeting that threshold before cycle 195: it appears in cycles 191, 192, 193, and 194, and those entries show repeated deferrals rather than resolution (`docs/state.json:3241-3294`). The cycle-195 worklog nevertheless records the cycle-194 finding as `**DEFERRED**` again (`docs/worklog/2026-03-08/193800-hundred-ninety-fifth-orchestrator-cycle.md:16-20`), and the journal explicitly says “Per step 0.6, it must be actioned or dropped,” then concludes that the three-bucket model is “sufficient,” while dispatching #804 only to prevent a 0/0/0 count-entry mistake (`docs/journal/2026-03-08.md:482-492`).

   That is not actioning the escalated finding, and it is not clearly dropping it either. It is a rationale for continuing the status quo while presenting that choice as satisfaction of the escalation rule. The code backs up why #804 is orthogonal: `process-review` still only models `actioned`, `deferred`, and `ignored` in both its CLI and serialized history entry (`tools/rust/crates/process-review/src/main.rs:23-50`). The mechanical validation dispatch addresses bad count totals, not the chronic complaint that the process lacks a machine-readable way to distinguish “dispatched,” “partially actioned,” and “truly actioned.”

   The next-cycle commitments themselves are concrete and verifiable (`docs/journal/2026-03-08.md:488-492`), but they do not resolve the escalated category the journal says cannot be left in limbo.

   **Recommendation:** either explicitly drop the disposition-overstatement concern and document why the review agent/spec should stop flagging it, or action it structurally by extending the canonical review-history model/tooling to represent the finer states being used in prose. What should not happen is citing step 0.6 and then recording another de facto deferral.

But the cycle still told a cleaner story than the repository supports. That is strong evidence of continued process self-exoneration, just not total denial.
