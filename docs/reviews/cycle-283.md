## 2. [worklog-accuracy] The chronic recalibration was marked runtime_verified before a cycle had actually proven zero factual worklog errors

**File**: docs/journal/2026-03-16.md:386-388
**Evidence**: Cycle 283 changes the `worklog-accuracy` chronic entry to `chosen_path: "recalibrate"` with `verification_cycle: 283`, which is the numeric form the invariants treat as runtime-verified. But the justification for recalibration was supposed to separate timing-inherent drift from factual mistakes. Cycle 282's own review still recorded factual worklog-accuracy defects, not just timing drift: a wrong audit-repository link and incomplete pipeline-failure reporting (`docs/reviews/cycle-282.md:3-19`). The cycle 283 journal then defers the actual proof into the future — `Regression check cycle 284: verify recalibration holds` (`docs/journal/2026-03-16.md:386-388`). That means the repository accepted the theory and stamped it runtime-verified in the same cycle before any subsequent review had confirmed the new zero-factual-errors standard. It collapses the distinction between "recalibration accepted" and "runtime proof observed."
**Recommendation**: Keep recalibrated chronic entries in a pending or explanatory state until a later review artifact demonstrates the new criterion was met; only then convert `verification_cycle` to a numeric runtime-verified value.

## 3. [state-integrity] The clean-cycle counter was incremented on an intermediate PASS that the actual complete-state commit immediately broke

**File**: docs/state.json:4364-4371
**Evidence**: Step `C5.5` on issue `#1382` says the final pipeline gate passed, and step `C5.6` increments `clean_cycle_counter` from `0 -> 1` on that basis. Running `bash tools/pipeline-check --json` against the clean-cycle commit `653d1dd` reproduces that PASS. But the very next cycle-tagged state commit, `state(record-dispatch): #1384 dispatched [cycle 283]` (`b3e0872`), moves `cycle_phase.phase` to `complete` and updates `copilot_metrics` for the new in-flight review. A fresh `bash tools/pipeline-check --json` on that commit fails with blocking `state-invariants: 14/15`. `bash tools/state-invariants` on the same commit shows the exact regression: `cycle_phase.phase is complete but cycle_phase.completed_at is missing`. **Recommendation**: Gate `clean_cycle_counter` increments on the post-record-dispatch `complete` state, or make the record-dispatch/close-out path write `cycle_phase.completed_at` before the clean-cycle decision is committed. A clean cycle that disappears at step `C7` should not count toward the stabilization total.
