## 2. [code-quality] `check-doc-pr` makes phased-resumption doc PRs structurally impossible once later cycle receipts land

**File**: tools/rust/crates/check-doc-pr/src/main.rs:795-860
**Evidence**: `check_receipt_completeness()` always loads the canonical receipt set by running `tools/cycle-receipts --cycle N --repo-root ROOT --json` against the current repository state, then fails if any of those receipts are absent from the worklog table. There is no scoping by PR base commit, doc-dispatch timestamp, or phase boundary. That means a docs PR created from the Phase A snapshot can become invalid the moment Phase B/C adds more cycle-tagged commits on `master`. Cycle 232 demonstrates the flaw directly: the canonical cycle-232 receipt set now includes `305ab7f` (`state(cycle-phase): close_out`) and `ccfc860` (`state(process-merge): PR #1088 merged`), but the docs task issue [#1089](https://github.com/EvaLok/schema-org-json-ld/issues/1089) and doc PR [#1090](https://github.com/EvaLok/schema-org-json-ld/pull/1090) were created before those commits existed. The journal then described the result as an “unfixable receipt gap” (`docs/journal/2026-03-12.md:27-38`), but that gap is being created by the validator’s current design, not merely exposed by it.
**Recommendation**: Freeze the expected receipt set at documentation-dispatch time (or derive it from the PR base) and validate against that frozen set. Add an integration-style regression that opens a docs PR from Phase A, lands additional cycle-tagged commits in Phase B/C, and proves `check-doc-pr` still distinguishes structural timing drift from an actually incomplete receipt table.
